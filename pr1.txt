Optimiser — Prolog GitHub Agent Program Requirements

Overview

Optimiser is a Prolog program transformation and optimisation system designed to convert recursive, nondeterministic, and structurally complex Prolog programs into simplified deterministic symbolic computations.

The system combines:

* memoisation,
* data unfolding,
* predicate simplification,
* subterm-address indexing,
* indexical optimisation,
* Gaussian elimination–based formula discovery,
* recursive loop conversion,
* and symbolic “manual neuronet” optimisation.

The optimiser is intended to:

* reduce unnecessary recursion,
* minimise repeated computation,
* convert recursive structures into formulas,
* optimise multidimensional data traversal,
* and produce simplified executable symbolic programs.

The optimiser operates as a multistage transformation pipeline.

⸻

Core Goals

The optimiser shall:

1. Simplify recursive predicates into formulas where possible.
2. Replace repeated recursive computation with memoised reuse.
3. Convert nested recursive traversal into deterministic index loops.
4. Discover mathematical patterns in recursive computations.
5. Use Gaussian elimination to infer formulas from sequences.
6. Reduce unnecessary enumeration and backtracking.
7. Construct only the outputs actually required.
8. Preserve semantic equivalence of the original program whenever possible.
9. Produce simplified transformed Prolog output.

⸻

Pipeline Stages

Stage 1 — Parsing and Structural Analysis

The system shall:

* parse Prolog source files,
* build internal representations of predicates,
* identify:
    * recursive predicates,
    * enumerators,
    * accumulators,
    * generators,
    * reusable variables,
    * deterministic predicates,
    * list-processing patterns,
    * matrix-processing patterns,
    * and nested subterm traversals.

Example internal representation:

ir_clause(Name, Args, Body)

⸻

Stage 2 — Data Unfolding and Memoisation

Purpose

This stage simplifies computation hierarchies by unfolding reusable computations from the base upward.

Requirements

The optimiser shall apply:

* base-up data unfolding,
* memoisation,
* and formula simplification.

Base-Up Unfolding

Simpler predicates shall be unfolded into higher-level predicates wherever beneficial.

Example:

inc(X,Y):-Y is X+1.
double(X,Y):-Y is X*2.

may unfold into:

combined(X,Y):-Y is (X+1)*2.

⸻

Memoisation

The optimiser shall:

* detect repeated subcomputations,
* store reusable results,
* reload previously computed values instead of recomputing them.

Example:

fib(N,F)

may become:

memo_fib(N,F)

⸻

Variable Reuse Optimisation

If a variable is reused later:

* the optimiser shall ensure it is available beforehand,
* loaded directly,
* or memoised in the optimal position.

⸻

Predicate-to-Formula Simplification

Recursive or iterative predicates should be replaced by formulas whenever possible.

Example:

sum_to_n(N,S)

becomes:

S is N*(N+1)//2

⸻

Stage 3 — Enumerator Optimisation

The optimiser shall distinguish between:

* created enumerators,
* removed enumerators,
* retained enumerators.

Created Enumerators

Enumerators introduced during unfolding or loop conversion.

Removed Enumerators

Enumerators made unnecessary after simplification.

Retained Enumerators

Enumerators still required for output generation.

Enumerator optimisation is separate from indexical optimisation.

⸻

Stage 4 — Indexical Optimisation

Purpose

Optimise computations using symbolic index relationships instead of exhaustive traversal.

Requirements

The optimiser shall:

* record list indices,
* matrix indices,
* tree indices,
* and nested subterm positions.

It shall relate input addresses directly to required outputs.

⸻

Address Representation

Nested structures shall use multidimensional addresses.

Example:

addr([2,1], X)

meaning:

Matrix[2][1]

⸻

Subterm Address Predicate

The optimiser shall support:

subterm_with_address(Term, Address, Subterm)

Example:

subterm_with_address([[a,b],[c,d]], [2,1], c).

⸻

Stage 5 — Recursive Index Loop Optimisation

Purpose

Convert recursive structural traversals into indexed loops.

Requirements

The optimiser shall detect recursively discovered nested structures and transform them into explicit index loops.

Example recursive traversal:

walk(Tree,X)

may become:

index_walk(Addresses,Outputs)

⸻

Subterm-Index Looping

The optimiser shall construct only required outputs.

Instead of:

process_all(Matrix,Out)

it may generate:

process_needed_indices(Matrix,Needed,Out)

This reduces unnecessary traversal.

⸻

Stage 6 — Gaussian Elimination Formula Discovery

Purpose

Detect numeric and recursive patterns and replace them with formulas.

Requirements

The optimiser shall use Gaussian elimination to infer formulas from generated sequences.

⸻

Sequence Detection

The optimiser shall detect:

* arithmetic sequences,
* stepped sequences,
* polynomial sequences,
* recursively generated numeric patterns.

Examples:

1,2,3,4,...
1,3,5,7,...
1,4,9,16,...

⸻

Formula Inference

The optimiser shall infer formulas such as:

A(N)=N
A(N)=2*N-1
A(N)=N*N

⸻

Sum Formula Discovery

The optimiser shall infer closed-form summation formulas.

Example:

sum(1..N)

becomes:

S=N*(N+1)/2

⸻

Recursive Sum Optimisation

If data unfolding across recursive list manipulations produces a detectable sequence, the optimiser shall replace the loop with a formula.

Example:

build_list(N,L),
sum_list(L,S)

may become:

S is N*(N+1)//2

⸻

General Sequence Support

The optimiser shall support:

* same-step sequences,
* differing-step sequences,
* recursively generated sequences,
* multidimensional index sequences.

⸻

Stage 7 — Shortcut and Splicing Optimisation

The optimiser shall detect values:

* used once,
* used only in comparisons,
* or directly forwarded.

It may splice computations directly into usage sites.

Example:

compare(f(X),Y)

instead of:

Temp=f(X),
compare(Temp,Y)

⸻

Stage 8 — Deterministic Loop Conversion

The optimiser may convert nondeterministic enumeration into deterministic loops where possible.

This includes:

* flattening nested recursion,
* replacing findall structures with loops,
* deterministic splicing,
* index-based traversal.

⸻

Stage 9 — Symbolic Manual Neuronet Optimisation

Purpose

Provide symbolic equivalents of neural optimisation.

Requirements

The optimiser shall:

* discover symbolic computation patterns,
* record input-output relationships,
* simplify data paths,
* minimise traversal,
* and generate direct symbolic computation paths.

Unlike neural networks, the optimiser shall use explicit symbolic transformations.

⸻

Input Requirements

The system shall accept:

* standard SWI-Prolog source files,
* recursive predicates,
* list-processing predicates,
* matrix-processing predicates,
* symbolic arithmetic predicates.

⸻

Output Requirements

The optimiser shall produce:

* transformed Prolog code,
* optimisation reports,
* detected formulas,
* index mappings,
* transformed loops,
* and optional optimisation traces.

⸻

Example Optimisation

Input

sum_to_n(0,0).
sum_to_n(N,S):-
    N>0,
    N1 is N-1,
    sum_to_n(N1,S1),
    S is S1+N.

Output

sum_to_n(N,S):-
    S is N*(N+1)//2.

⸻

Example Nested Index Optimisation

Input

nth1(I,Matrix,Row),
nth1(J,Row,X).

Output

addr([I,J],X).

⸻

Example Recursive Loop Optimisation

Input

member(X,List),
process(X,Y).

Output

for_index(List,I,X),
process(X,Y).
